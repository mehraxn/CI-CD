# Release Automation, Communication, and Audit

## From Event to Evidence

```text
Release event
    ↓
Pipeline executes
    ↓
Deployment record created
    ↓
Verification result recorded
    ↓
Stakeholders notified
    ↓
Audit evidence retained
```

Release automation orchestrates triggers, candidate selection, deployment, verification, records, and notifications. Triggers may be tags, manual workflows, schedules, or approved promotion events. ChatOps, tickets, change-management systems, status pages, and release dashboards are integrations, not sources of truth by themselves.

A useful release or deployment notification includes version, environment, immutable artifact identity, change summary, start/completion time, result, known issues, owner, and rollback status. A deployment record says what changed in an environment; a release record says what became available. Both should connect approver identity, pipeline run, timestamp, failure reason, and retained evidence.

Communication must avoid tokens, credentials, sensitive internal addresses, and unnecessary personal data. Deduplicate messages and route by audience to prevent notification fatigue. The pipeline history should be authoritative; manual records that disagree with it must be reconciled.

Release orchestration sequences tools but should not hide ownership. Tag-based automation must verify the tag and artifact relationship. Manual workflows should use validated inputs and record the actor. Scheduled releases need a policy for candidate selection and for postponing a failed candidate.

Stakeholders need different views: operators need artifact, environment, timing, health, and recovery status; users need impact and known issues; auditors need immutable identity, evidence, approver, and retention. A status page communicates customer-visible state but must not expose internal credentials or unnecessary topology.

ChatOps can request or report actions in chat, while ticket and change-management integration can attach evidence to an existing process. Neither should allow an unreviewed message to bypass pipeline authorization. A release dashboard should link to source, run, artifact, deployment, verification, and rollback records rather than copying inconsistent values.

Audit retention must last long enough for incident investigation and compliance. Failure notifications are as important as success messages: they state the failed stage, impact, owner, next update, and whether rollback occurred. Notification fatigue is controlled with one correlation key per release and state transitions rather than repeated raw logs.

## Common Mistakes

- Notifications without context or with sensitive values.
- No artifact-to-release connection or failure notice.
- Duplicate noise and missing retention.
- Manual records disagreeing with pipeline history.
- No owner for failed releases.

## Practical Exercise

Design a TaskOps deployment record and notification containing safe fields only. Specify source systems, recipients, deduplication key, failure message, rollback update, owner, and retention. Do not send it.

## Knowledge Check

1. Release record versus deployment record?
2. What identity belongs in both?
3. Why is pipeline history preferable to a manual spreadsheet?
4. How is notification fatigue reduced?
5. What release automation is absent here?

<details><summary>View answers</summary>

1. Deployment records environment change; release records availability.
2. Exact immutable artifact identity plus source revision.
3. It is generated by execution and is less prone to manual disagreement.
4. Deduplicate, route by audience, and send actionable context.
5. Formal release records, notifications, tag releases, dashboards, and integrated change management.

</details>
