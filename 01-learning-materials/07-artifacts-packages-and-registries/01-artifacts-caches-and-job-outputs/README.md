# Artifacts, Caches, and Job Outputs

## Nothing Survives a Runner

Jobs run in isolated, temporary workspaces. A file created in one job does not exist in the next; when the run finishes, the workspace is destroyed. Everything that must outlive that boundary needs one of three explicit mechanisms:

```text
Small metadata:
Use job or step outputs.

Files required by another job:
Use pipeline artifacts or an external artifact repository.

Recreatable dependency data:
Use a cache.
```

**Job and step outputs** carry strings — a version, an image name, a flag (real usage: KubeOps image-release passes its computed image name via `$GITHUB_OUTPUT`, covered in [Variables and Outputs](../../04-pipeline-as-code-and-platforms/03-variables-contexts-expressions-and-outputs/)). **Pipeline artifacts** carry files. **Caches** carry recreatable speed-ups. Confusing them produces the classic failures: files stuffed into outputs, release evidence entrusted to a cache that eviction silently deletes.

## Kinds of Artifacts

- **Build artifacts** — the packaged output itself.
- **Test artifacts** — reports, coverage files.
- **Diagnostic artifacts** — logs, screenshots, crash dumps; most valuable *on failure*, which is why evidence-uploading steps often run with `if: always()` or `if: failure()`.
- **Release artifacts** — verified outputs kept for delivery, rollback, and audit.
- **Security reports** — scan results preserved as evidence.

## Moving Files Between Jobs

A conceptual GitHub Actions example (this repository does not currently use the artifact actions):

```yaml
jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Create build output
        run: |
          mkdir -p dist
          echo "application package" > dist/app.txt

      - name: Upload build artifact
        uses: actions/upload-artifact@v4
        with:
          name: application-package
          path: dist/

  verify:
    needs: build
    runs-on: ubuntu-latest

    steps:
      - name: Download build artifact
        uses: actions/download-artifact@v4
        with:
          name: application-package
          path: dist/

      - name: Inspect artifact
        run: ls -la dist/
```

Reading the mechanics:

- `verify` runs on a **different runner** with a fresh workspace — nothing is shared automatically, which is why the **download step** exists at all.
- The artifact **name** (`application-package`) is the lookup key between jobs; the **path** says what to package and where to unpack it.
- `needs: build` guarantees ordering — downloading an artifact that has not been uploaded yet fails.
- This differs from caching in intent and guarantees: the artifact *must* exist for `verify` to work; a cache may be absent and the job must cope.

Artifacts are typically **compressed** on upload, have **size limits** and **retention/expiration** windows (platform- and configuration-dependent), and remain downloadable from the run page **after the pipeline completes** — that is what makes them evidence. **Integrity** deserves thought: record **checksums** for anything security-relevant, and validate the origin of anything you download. **Permissions** follow the run: whoever can read the run can usually read its artifacts, so never upload secret-bearing files.

## Comparison

| Mechanism | Best for | Required to be reproducible? | Typical lifetime |
|-----------|----------|------------------------------|------------------|
| Job output | Small metadata | Generated by workflow | One workflow run |
| Artifact | Build and test files | Often important | Configurable |
| Cache | Dependencies and reusable build data | No; should be recreatable | Temporary |
| Registry | Versioned packages and images | Yes for releases | Long-lived |

## Common Mistakes

- Passing large files through outputs.
- Treating caches as release evidence.
- Assuming artifacts exist forever — retention expires them.
- Uploading secret-bearing files.
- Uploading an entire workspace when one directory would do.
- Unclear artifact names (`output`, `files`) that mean nothing a week later.
- Failing to preserve test evidence after failures — exactly when it is needed.
- Depending on expired artifacts in later re-runs.
- Downloading an artifact without validating its origin.
- Publishing unverified output to long-lived storage.

## Existing Repository Evidence

The repository currently moves *no files* between jobs — a deliberate architecture worth examining. In [TaskOps CI](../../../Projects/1_project/taskops-cicd/.github/workflows/ci.yml), the `docker` job does not receive files from `quality`; it checks out source and builds independently, using the BuildKit **cache** (`type=gha`) for speed. The built image travels between workflows via the **registry** ([TaskOps CD](../../../Projects/1_project/taskops-cicd/.github/workflows/cd.yml) pushes to GHCR and the server pulls from GHCR) — registry-as-artifact-store, the container-native pattern. Job **outputs** are used in [KubeOps image release](../../../Projects/2_project/kubeops-gitops/.github/workflows/image-release.yml) (`steps.img.outputs.image`). Pipeline artifacts (`actions/upload-artifact`) are not demonstrated; test reports and scan results currently live only in job logs — a real gap this lesson's exercise explores.

## Knowledge Check

1. Why does a second job need an explicit download step?
2. What belongs in a job output versus an artifact?
3. Why must a cache never be the only home of release evidence?
4. Why should diagnostic artifacts upload even when the job fails?
5. How does this repository move its main build output between workflows?
6. What is the risk of uploading a whole workspace as an artifact?

<details>
<summary>View answers</summary>

1. Jobs run on isolated runners with fresh workspaces; nothing transfers unless explicitly uploaded and downloaded.
2. Outputs carry small metadata strings (versions, names, flags); artifacts carry files.
3. Caches may be evicted or recreated at any time; evidence and release content require guaranteed preservation.
4. Failure is when diagnosis is needed — evidence steps should run with `if: always()` or `if: failure()` so the proof survives.
5. Via the registry: images are pushed to GHCR and pulled by the deployment target — registry-as-artifact-store.
6. Wasted storage and time, and a far higher chance of accidentally including secret-bearing or irrelevant files.
